# new-feature Phase 3: 設計書の観点

## 設計書

- `docs/design_analysis/new_feature/<yyyymmdd>_<topic>/design/<topic>_feature_design.md`

## 必須観点

- 機能追加に必要な責務分割
- UI / データ / サービス / 永続化の追加点
- 拡張ポイントと将来の派生機能への耐性
- 失敗時動作とデフォルト挙動
- 既存類似ロジックとの抽象化・共通化方針
- 新規関数の配置先を責務で判断し、呼び出し箇所に近いだけの場当たり配置を避ける方針
- モジュール直下のグローバル関数を安易に増やさず、責務に応じたクラス/メソッドへ収める方針
- `getattr` / `setattr` / `Any` / `Dict` を使う必要がある場合の局所化方針
- 旧経路を並存させない単一路線の設計方針

## レビュー観点

- 機能追加の境界が曖昧になっていないか
- 既存アーキテクチャへ無理なく収まるか
- 類似責務の重複追加や不要な互換レイヤーに依存していないか
- 関数配置が責務に沿っており、場当たり的な helper 追加で境界を曖昧にしていないか
- 将来拡張の余地を潰していないか
