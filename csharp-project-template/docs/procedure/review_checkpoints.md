# レビュー観点（詳細）

## 1. 仕様整合

- ドキュメント仕様と実装挙動に差分がないか
- 仕様未記載の制約がコードに埋め込まれていないか

## 2. 設計品質

- 責務分離（SRP）が保たれているか
- 既存設計（レイヤー構造、Command/Store、サービス境界など）に矛盾しないか
- 関連 ADR が存在する場合、変更がその判断に反していないか
- 不要な重複実装（DRY違反）がないか
- 類似ロジックがある場合に、重複追加ではなく抽象化・共通化が優先されているか
- 抽象化・共通化の結果が型安全性や責務境界を壊していないか
- 新規メソッド/関数の配置先が、特定クラス固有 / 抽象化層 / 汎用層のいずれとして妥当か
- トップレベル関数相当や責務の曖昧な `static` helper を原則増やさず、責務に応じたクラス/メソッドへ収まっているか
- フレームワーク契約や既存公開 API 等の明示要件で例外を採る場合、理由と境界が記録されているか
- API/命名/責務分離の一貫性が保たれているか

## 3. 安全性・保守性

- 例外を握りつぶしていないか
- `reflection` / `dynamic` / `object` / `Dictionary<string, object?>` が抽象化・共通化・汎用化のための局所利用に留まっているか
- 不要なフォールバックで不具合を隠していないか
- プロジェクト内で閉じる仕様不一致をフォールバックで吸収せず、例外や明示的エラーとして顕在化しているか
- ログに原因追跡情報が残るか

## 4. パフォーマンス

- UI スレッドをブロックする I/O がないか
- パフォーマンス制約（タイムバジェット等）を考慮しているか
- 不要な再計算やメモリ確保がないか

## 5. 互換性・回帰

- 明示された互換要件または外部契約を壊していないか
- 変更対象外の既存 API/UI に意図しない振る舞い変更を混入していないか
- 要求されていない後方互換レイヤーや旧経路を残していないか
- 既存テストケースの意味が変わっていないか

## 6. テスト

- 変更点を直接検証するテストがあるか
- 境界値・失敗系が網羅されているか
- 統合テスト条件を満たしているか

## 7. ドキュメント

- 設計書とレビュー書の対応ステータスが一致しているか
- ADR で閉じるべき横断判断を、案件文書だけに埋め込んでいないか
- 関連コンポーネント文書へ反映済みか
- 記載不備/不整合を見つけた場合にその場で修正したか
