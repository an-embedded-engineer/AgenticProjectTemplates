# コーディングルール

## 基本方針

- 本プロジェクトは、C# コンパイラと .NET Analyzer を用いた静的解析で型安全性・コード品質を担保することを実装の基盤とする
- 必須ルール 2/3/4/6 は、この型安全性・品質方針から導出されたルールとして運用する
- 完了条件として `dotnet build --warnaserrors` の警告・エラー 0 件を必須とする

## 必須ルール

1. XML Doc コメント
   - public なクラス/メソッド/プロパティに XML Doc コメント (`/// <summary>`) を付ける
   - コメント・XML Doc は日本語で記述する
   - メソッド内部は処理ブロックごとに短いコメントを付ける
   - 根拠: 仕様意図と責務境界をコード上に残し、レビュー品質と保守性を担保するため
2. Nullable 参照型
   - プロジェクトで `<Nullable>enable</Nullable>` を有効にする
   - `T?` を使う場合は必ず null チェックとセットで使用する
   - `object` 型や `dynamic` 型は I/O 境界以外で使わない
   - 根拠: コンパイラによる null 安全性検査を最大化し、NullReferenceException を未然に防ぐため
3. リフレクション・動的アクセス制限
   - `reflection` / `dynamic` / `ExpandoObject` を避ける
   - 根拠: リファクタリング耐性と型推論精度を維持し、静的解析の検出漏れを防ぐため
4. 型付きデータモデル
   - `Dictionary<string, object>` での汎用的なデータ受け渡しを避ける
   - `record` / `class` / 専用の DTO を使用する
   - 根拠: データ契約を型として明示し、キー欠落/型不一致をコンパイル時に検知できるようにするため
5. 例外処理
   - 例外は握りつぶさない
   - 失敗時は原因が追跡できる情報をログへ残す
   - 根拠: 障害解析の再現性を確保し、障害の潜在化を防ぐため
6. フォールバック
   - 不必要なフォールバックを実装しない
   - フォールバックが必要な場合は条件と理由をコメントで明示する
   - 根拠: 問題の早期顕在化を優先し、不整合の隠蔽や複雑性の増加を防ぐため

## 設計原則

- DRY: 同じ責務の重複ロジックを排除する
  - 根拠: 変更点の一元化により不整合と修正漏れを抑止するため
- SOLID:
  - SRP: 1 クラス/メソッド 1 責務を維持する
  - OCP: 既存修正より拡張で対応できる設計を優先する
  - LSP: 派生型は基底型契約を破らない
  - ISP: 使わない依存を強制しない
  - DIP: 具体実装より抽象に依存する
  - 根拠: 変更容易性、テスト容易性、責務分離を維持して長期保守コストを下げるため
- デザインパターン: 必要に応じて Strategy, Factory, Command 等を採用する
  - 根拠: 可変点の明確化と実装の一貫性確保に有効なため
