# Phase 2 共通: 方針・設計レビュー

## 実施内容

1. workflow 固有の観点で `design/` 文書を作成する
2. `design/` 文書には、実装前に合意すべき判断材料を必須章として含める
   - 背景・要求・完了条件
   - 対象範囲と非対象
   - 採用案 / 不採用案 / 判断理由
   - before / after
   - 影響範囲
   - 設計方針
   - 互換性・移行方針
   - 恒久ドキュメント更新予定先
   - テスト・ユーザ確認観点
   - リスクと follow-up
3. `meta.md` の `design_status` を `draft` に更新する
4. 設計書と `meta.md` をコミットする
5. ユーザへレビュー依頼準備完了を報告し、レビュー完了まで待機する

> STOP: ユーザにレビュー依頼の準備完了を報告する。ユーザがレビュー担当 Agent を起動し、設計レビューが完了するまで待機する。

> 補足: `claude-review-automation` を使う場合、準備完了の報告後はユーザの追加承認を待たずに Codex が Claude review を起動してよい。監視は `15s -> 30s -> 60s` の backoff を使い、5 分程度の `thinking` は通常範囲として扱う。10 分程度進捗が見えない場合のみ、再起動前にユーザへ対応方針を確認する。Phase 区切りでは短い状態要約を残し、利用可能なら `/compact` を実行してよい。

6. レビュー文書作成完了後、`design_status` を `in_review` に更新して指摘を反映する
7. 未対応指摘がゼロになったら `design_status` を `done` に更新し、反映結果をコミットする

> STOP: Phase 2 完了をユーザに報告し、Phase 3 への進行承認を待つ。承認なしに Phase 3 に進んではならない。

## 設計時の必須判断

- 類似ロジックが既存にある場合は、新規重複実装より抽象化・共通化を優先し、採否理由を設計書へ残す
- 関数追加時は、特定クラス固有 / 抽象化層 / 汎用層のどこへ置くべきかを精査し、呼び出し箇所に近いだけの配置を避ける
- モジュール直下のグローバル関数は原則追加せず、責務に応じたクラスとメソッド種別を設計する
- フレームワーク契約や既存公開 API 等の明示要件で例外を採る場合は、理由、境界、採用期間を設計書へ残す
- `getattr` / `setattr` / `Any` / `Dict` を使う場合は、抽象化・共通化・汎用化のための局所利用に限定し、理由と型境界を設計書へ明記する
- 明示要求または外部契約がない限り、不要な分岐や代替経路を残さない
- プロジェクト内で閉じる仕様不一致はフォールバック前提にせず、例外や明示的エラーハンドリング方針を設計へ記載する
- 大規模 planning が必要な場合は、WBS 分解用 workflow へ切り出す

## レビュー担当への依頼

- レビュー担当 Agent は `ai-review-response-workflow` skill に従う
- レビュー観点詳細は `ai-review-response-workflow` skill に同梱された `references/procedure/review_checkpoints.md` を参照する

## 完了条件

- 設計書が要求・範囲・設計・検証・docs 更新予定を一体で説明している
- `design_status=done`
- レビュー前後の更新がコミットされている

## 承認ゲート

- 設計レビュー完了後、ユーザ承認なしに Phase 3 へ進まない
