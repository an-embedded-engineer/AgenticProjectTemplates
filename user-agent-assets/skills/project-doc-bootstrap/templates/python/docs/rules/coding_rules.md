# コーディングルール

## 基本方針

- 本プロジェクトは、Pylance/Pyright を用いた静的型検証で型安全性を担保することを実装の基盤とする
- 必須ルール 2/3/4 は、この型安全性方針から導出されたルールとして運用する
- 必須ルール 6/7 は、単一路線の設計維持と不整合の早期顕在化を補助するルールとして運用する
- 必須ルール 8/9 は、責務境界と所属の明確化を補助するルールとして運用する
- 類似ロジックを見つけた場合は、重複実装より抽象化・共通化を優先して検討する
- 関数やロジックの配置は、呼び出し箇所との近さではなく責務の所属と再利用性で判断する
- 完了条件として `./.venv/bin/pyright --outputjson > pylance_error.json` のエラー 0 件を必須とする

## 必須ルール

1. コメント/Docstring
   - クラス/メソッド/関数/フィールド直前に 1 行コメントを付ける
   - クラス/メソッド先頭に日本語 Docstring を付ける
   - メソッド内部は処理ブロックごとに短いコメントを付ける
   - 根拠: 仕様意図と責務境界をコード上に残し、レビュー品質と保守性を担保するため
2. 型ヒント
   - フィールド/引数/戻り値/ローカル変数に型注釈を付ける
   - `Optional` は `None` ガードとセットで使用する
   - `Any` / `dict[str, Any]` / `Dict[...]` は I/O 境界以外では原則使わない
   - 抽象化・共通化・汎用化のために必要な場合のみ、局所的に閉じた範囲で使用し、理由と境界をコードまたは文書に残す
   - 根拠: Pylance/Pyright による事前検出を最大化し、実行時不具合を未然に防ぐため
3. 文字列ベースアクセス禁止
   - `getattr` / `setattr` / `__dict__` を避ける
   - 抽象化・共通化・汎用化のために重複実装を防ぐ局所コードでのみ使用可とし、対象型と利用理由を明示する
   - 根拠: リファクタリング耐性と型推論精度を維持し、静的解析の検出漏れを防ぐため
4. 生 `dict` 受け渡し禁止
   - `dataclass` / `TypedDict` / Pydantic を使用する
   - 抽象化・共通化・汎用化のために辞書構造を使う場合も、そのスコープ内で閉じ、公開境界へそのまま渡さない
   - 根拠: データ契約を型として明示し、キー欠落/型不一致を静的に検知できるようにするため
5. 例外処理
   - 例外は握りつぶさない
   - 失敗時は原因が追跡できる情報をログへ残す
   - 根拠: 障害解析の再現性を確保し、障害の潜在化を防ぐため
6. 後方互換
   - 明示的に指示されない限り、旧経路/旧 API/旧データ解釈を残す後方互換レイヤーは実装しない
   - 置き換え後は呼び出し側・テスト・文書を更新し、最適な経路 1 つに収束させる
   - 根拠: 開発段階では最適設計の探索を優先し、複数経路の併存による可読性・保守性低下を防ぐため
7. フォールバック
   - 不必要なフォールバックを実装しない
   - プロジェクト内で閉じる仕様不一致はフォールバックで吸収せず、例外送出や明示的なエラーハンドリングで顕在化する
   - フォールバックが必要な場合は条件と理由をコメントで明示する
   - 根拠: 問題の早期顕在化を優先し、不整合の隠蔽や複雑性の増加を防ぐため
8. 関数の配置
   - 関数追加時は、その責務が特定クラス固有か、より抽象的か、汎用的かを先に判断する
   - 変更範囲を局所化したいだけの理由で、呼び出し箇所に近い場所へ安易に追加しない
   - 類似関数が既存にある場合は、新設より統合・再配置・共通化を優先する
   - 根拠: 呼び出し元近傍への場当たり追加は類似コードの増殖を招き、責務境界と保守性を悪化させるため
9. モジュール関数
   - モジュール直下のグローバル関数は原則追加しない
   - 原則としてクラスを定義し、責務に応じてインスタンスメソッド / クラスメソッド / スタティックメソッドを選択する
   - フレームワーク契約や既存公開 API 等の明示要件がある場合のみ例外とし、理由を文書またはコメントに残す
   - 根拠: 所属の分かりやすさ、import の単純化、責務単位での変更容易性を維持するため

## 設計原則

- DRY: 同じ責務の重複ロジックを排除する
  - 類似ロジックを見つけたら、コピーによる局所対応より先に抽象化・共通化を検討する
  - 根拠: 変更点の一元化により不整合と修正漏れを抑止するため
- SOLID:
  - SRP: 1 クラス/メソッド 1 責務を維持する
  - OCP: 既存修正より拡張で対応できる設計を優先する
  - LSP: 派生型は基底型契約を破らない
  - ISP: 使わない依存を強制しない
  - DIP: 具体実装より抽象に依存する
  - 根拠: 変更容易性、テスト容易性、責務分離を維持して長期保守コストを下げるため
- デザインパターン: 必要に応じて Strategy, Factory, Command 等を採用する
  - 根拠: 可変点の明確化と実装の一貫性確保に有効なため
