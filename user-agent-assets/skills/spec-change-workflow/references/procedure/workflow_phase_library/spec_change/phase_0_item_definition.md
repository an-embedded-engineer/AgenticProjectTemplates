# spec-change Phase 0: 要求整理

## 起票先

- `docs/todo/todo.md`

## 実施内容

1. 変更対象の既存仕様、期待する新仕様、非対象を明確化する
2. 影響コンポーネントと変更される契約を列挙する
3. 受け入れ条件を `docs/todo/todo.md` に記録する
4. 既存 issue が背景にある場合は `docs/issues/` への参照だけ残し、正本は `todo.md` とする
5. Phase 0 の成果をコミットする

## 調査観点

- どの画面・操作・データ契約が変わるか
- 既存仕様との互換性に破壊的変更があるか
- 更新すべき恒久ドキュメントはどこか

## 完了条件

- 目的、範囲、受け入れ条件が `docs/todo/todo.md` で追跡可能
- 影響コンポーネントが定義されている
- Phase 0 成果がコミットされている
