# spec-change Phase 2: 方針・設計の観点

## 設計書

- `docs/design_analysis/spec_change/<yyyymmdd>_<topic>/design/<topic>_design.md`

## 必須観点

- 変更理由と現行仕様との差分
- 受け入れ条件ごとの対応方針
- 対象範囲と非対象
- 採用案 / 不採用案 / 判断理由
- before / after の仕様差分
- UI / API / データモデル / ドメインルールの変更点
- 影響コンポーネント別の変更範囲
- コンポーネント責務と依存方向
- 既存データとの整合、移行有無、デフォルト動作
- 類似既存ロジックとの抽象化・共通化方針
- 互換要件がある場合の条件と、不要な経路を削除する方針
- プロジェクト内で閉じる仕様不一致に対する例外・エラーハンドリング方針
- 恒久ドキュメント更新予定先
- テスト観点とユーザ確認シナリオ
- リスクと follow-up

## レビュー観点

- 受け入れ条件が漏れなく設計へ落ちているか
- 仕様変更の境界が曖昧になっていないか
- 仕様差分が実装可能な粒度で定義されているか
- 境界条件と副作用が設計に落ちているか
- 類似ロジックの重複追加ではなく、共通化方針が妥当か
- 過剰な互換レイヤーや不要フォールバックを前提にした設計になっていないか
- 関連ドキュメント更新先が明記されているか
