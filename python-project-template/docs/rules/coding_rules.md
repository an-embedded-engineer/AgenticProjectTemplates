# コーディングルール

## 基本方針

- 本プロジェクトは、Pylance/Pyright を用いた静的型検証で型安全性を担保することを実装の基盤とする
- 必須ルール 2/3/4/6 は、この型安全性方針から導出されたルールとして運用する
- 完了条件として `./.venv/bin/pyright --outputjson > pylance_error.json` のエラー 0 件を必須とする

## 必須ルール

1. コメント/Docstring
   - クラス/メソッド/関数/フィールド直前に 1 行コメントを付ける
   - クラス/メソッド先頭に日本語 Docstring を付ける
   - メソッド内部は処理ブロックごとに短いコメントを付ける
   - 根拠: 仕様意図と責務境界をコード上に残し、レビュー品質と保守性を担保するため
2. 型ヒント
   - フィールド/引数/戻り値/ローカル変数に型注釈を付ける
   - `Optional` は `None` ガードとセットで使用する
   - `Any` は I/O 境界以外で使わない
   - 根拠: Pylance/Pyright による事前検出を最大化し、実行時不具合を未然に防ぐため
3. 文字列ベースアクセス禁止
   - `getattr` / `setattr` / `__dict__` を避ける
   - 根拠: リファクタリング耐性と型推論精度を維持し、静的解析の検出漏れを防ぐため
4. 生 `dict` 受け渡し禁止
   - `dataclass` / `TypedDict` / Pydantic を使用する
   - 根拠: データ契約を型として明示し、キー欠落/型不一致を静的に検知できるようにするため
5. 例外処理
   - 例外は握りつぶさない
   - 失敗時は原因が追跡できる情報をログへ残す
   - 根拠: 障害解析の再現性を確保し、障害の潜在化を防ぐため
6. フォールバック
   - 不必要なフォールバックを実装しない
   - フォールバックが必要な場合は条件と理由をコメントで明示する
   - 根拠: 問題の早期顕在化を優先し、不整合の隠蔽や複雑性の増加を防ぐため

## 設計原則

- DRY: 同じ責務の重複ロジックを排除する
  - 根拠: 変更点の一元化により不整合と修正漏れを抑止するため
- SOLID:
  - SRP: 1 クラス/メソッド 1 責務を維持する
  - OCP: 既存修正より拡張で対応できる設計を優先する
  - LSP: 派生型は基底型契約を破らない
  - ISP: 使わない依存を強制しない
  - DIP: 具体実装より抽象に依存する
  - 根拠: 変更容易性、テスト容易性、責務分離を維持して長期保守コストを下げるため
- デザインパターン: 必要に応じて Strategy, Factory, Command 等を採用する
  - 根拠: 可変点の明確化と実装の一貫性確保に有効なため
