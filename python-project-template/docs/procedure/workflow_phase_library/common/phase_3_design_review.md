# Phase 3 共通: 設計作成・設計レビュー

## 実施内容

1. workflow 固有の観点で `design/` 文書を作成する
2. `meta.md` の `design_status` を `draft` に更新する
3. 設計書と `meta.md` をコミットする
4. ユーザへレビュー依頼準備完了を報告し、レビュー完了まで待機する

> ⛔ **STOP**: ユーザにレビュー依頼の準備完了を報告する。ユーザがレビュー担当 Agent を起動し、設計レビューが完了するまで待機する。

> 補足: `claude-review-automation` を使う場合、準備完了の報告後はユーザの追加承認を待たずに Codex が Claude review を起動してよい。監視は `15s -> 30s -> 60s` の backoff を使い、5 分程度の `thinking` は通常範囲として扱う。10 分程度進捗が見えない場合のみ、再起動前にユーザへ対応方針を確認する。Phase 区切りでは短い状態要約を残し、利用可能なら `/compact` を実行してよい。

5. レビュー文書作成完了後、`design_status` を `in_review` に更新して指摘を反映する
6. 未対応指摘がゼロになったら `design_status` を `done` に更新し、反映結果をコミットする

> ⛔ **STOP**: Phase 3 完了をユーザに報告し、Phase 4 への進行承認を待つ。承認なしに Phase 4 に進んではならない。

## 設計時の必須判断

- 類似ロジックが既存にある場合は、新規重複実装より抽象化・共通化を優先し、採否理由を設計書へ残す
- `getattr` / `setattr` / `Any` / `Dict` を使う場合は、抽象化・共通化・汎用化のための局所利用に限定し、理由と型境界を設計書へ明記する
- 明示要求または外部契約がない限り、旧経路を残す後方互換設計を採らない
- プロジェクト内で閉じる仕様不一致はフォールバック前提にせず、例外や明示的エラーハンドリング方針を設計へ記載する

## レビュー担当への依頼

- レビュー担当 Agent は `docs/procedure/ai_review_response_workflow.md` に従う
- レビュー観点詳細は `docs/procedure/review_checkpoints.md` を参照する

## 完了条件

- 設計書が最新レビュー結果を反映している
- `design_status=done`
- レビュー前後の更新がコミットされている

## 承認ゲート

- 設計レビュー完了後、ユーザ承認なしに Phase 4 へ進まない
